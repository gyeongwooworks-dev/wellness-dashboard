# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-page, build-free analytics dashboard (Korean-language UI) that presents wearable-device data (steps, activity, sleep, heart rate, calories) to two internal audiences — marketing (segmentation/personas) and product (behavioral insights/retention) — from a common KPI bar. Source data is the Kaggle FitBit Fitness Tracker Data sample (35 users, two merged observation windows in 2016). See [docs/PRD.md](docs/PRD.md) for the full product spec (metric definitions, segment rules, section-by-section requirements) and [README.md](README.md) for deployment/data notes in Korean.

## Architecture

The entire app is one file, [index.html](index.html), with no build step, framework, or package manager:

- **`<style>` (lines ~10-125)**: design tokens as CSS custom properties (`--primary`, `--label`, spacing/radius scale) followed by component styles.
- **HTML body (lines ~127-263)**: static shell with empty containers (`#kpis`, `#badges`, chart `<canvas>` elements, `#personaCards`, `#tblBody`) that JS fills in at runtime. Sections are tagged with PRD IDs (`M1`-`M4` marketing, `P1`-`P5` product) matching [docs/PRD.md](docs/PRD.md) section 6.
- **`<script>` (lines 265-474)**:
  - Line 266 is a single-line `const DATA = {...}` — the entire pre-aggregated dataset inlined as JSON (no fetch, no server required). This is generated by [scripts/aggregate.py](scripts/aggregate.py) and copy-pasted in; it is also duplicated at [data/data_wellness.json](data/data_wellness.json).
  - Everything after that is plain DOM manipulation + Chart.js (loaded from CDN, `chart.umd.min.js`). No framework, no modules, no bundler.
  - `refresh()` is the central re-render function, triggered by the three global filter `<select>` elements (`segSel` = activity segment, `userSel` = individual user drilldown, `metricSel` = trend metric). It calls the per-chart `draw*` functions (`drawKpis`, `drawM1`, `drawRadar`, `drawPersonas`, `drawHour`, `drawTrend`, `drawScatter`, `drawCons`, `drawWeekly`, `drawHR`, `drawCoverage`, `renderTbl`).
  - Charts are stored in the `ch` object and explicitly `.destroy()`'d before redraw to avoid Chart.js leaks — follow this pattern when adding new charts.
  - `init()` at the bottom wires everything up and calls `refresh()` once on load.

Because there is no build step, **all JS/CSS changes are made directly in `index.html`** — there is no separate `.js`/`.css` source to edit and compile.

## Data pipeline

[scripts/aggregate.py](scripts/aggregate.py) is a one-off, non-idempotent-path script (hardcoded absolute input/output paths from the original authoring sandbox) that reads the raw FitBit CSV export and produces the aggregated JSON consumed by the dashboard. It is not run as part of any normal workflow — the aggregated output is already committed/inlined. Re-run it only when the underlying raw data changes, and expect to fix the `ROOT`/output paths at the top and bottom of the file first (they point to a sandbox path, not this repo). Key aggregation rules (also documented in [README.md](README.md)):

- Steps/calories/activity/hourly/heart-rate: two observation periods (2016-03-12–04-09 and 04-12–05-12) are **merged**.
- Sleep: only the second period (04-12–05-12, 24 users with `sleepDay` summaries) is used.
- Users: union of both periods = 35.
- Consistency (engagement proxy) = days recorded ÷ total observation window (62 days).
- Activity segments: power (avg steps ≥10,000) / steady (5,000–10,000) / sedentary (<5,000).
- Engagement segments: core (≥80% consistency) / casual (40–80%) / at-risk (<40%).

After regenerating, the new JSON must be manually pasted into the `const DATA=...` line in `index.html` — the file intentionally has no runtime fetch of `data/data_wellness.json`.

## Running locally

No build/install step. Serve the directory with any static server and open it (or just double-click `index.html`, since data is inlined and only fonts/Chart.js need network access):

```bash
python3 -m http.server 8000
# or
npx serve .
```

There are no tests, linters, or CI configured in this repo.

## Deployment

Deployed as a static site via GitHub Pages directly from this folder (root `index.html`, `.nojekyll` present to skip Jekyll processing). No CI/build step — pushing to `main` with Pages configured for `/ (root)` is sufficient. See [HANDOFF.md](HANDOFF.md) for the exact repo-creation/Pages-enablement command sequence used originally (via `gh` CLI), including confirmation checkpoints for creating a public repo and the first push.
